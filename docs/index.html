🔬 原理详解（面向电气工程师）

一、梅尔频谱：为什么是 80×T 的矩阵？

人耳对频率的感知是对数尺度而非线性尺度。1000Hz 和 1100Hz 的差距很容易分辨，但 10000Hz 和 10100Hz 几乎无法区分。

mel = 2595 · log10(1 + f / 700)

FastSpeech2 把语音分解为 80 个梅尔频带 × 时间帧 的二维矩阵：

  • 每一列是该时刻的「瞬时频谱」——相当于频谱分析仪的某一帧画面
  • 每一行是该频带的「音量包络」——相当于该频率的 RMS 电压随时间变化

类比：示波器看时域（一维波形），频谱仪看频域（一维频谱），语谱图看时频域（二维梅尔频谱）——这就是 FastSpeech2 输出的本质。

二、检查点 (.pdz) vs 纯权重 (state_dict)

PaddlePaddle 的 .pdz 文件是训练检查点（checkpoint），包含大量推理不需要的数据：

内容占比推理需要？
模型权重 (weight)~20%✅ 必须
Adam 优化器 1 阶矩 m~30%❌ 训练才要
Adam 优化器 2 阶矩 v~30%❌ 训练才要
学习率调度器状态~6%❌
Pickle 序列化开销~14%❌

Adam 优化器为什么占这么大？

Adam 为每一个可训练参数维护两个同形状的矩阵：

  • m（一阶矩）：梯度的指数移动平均，记录「参数一直在往哪个方向走」
  • v（二阶矩）：梯度平方的指数移动平均，记录「参数的变化有多剧烈」

如果模型有 N 个参数，Adam 额外存储 2N 个值——检查点因此是纯权重的 3 倍。

类比：检查点像一个包含所有草稿、批注、修订记录的 Word 文档，而 state_dict 只是最终干净的 PDF。推理只需要 PDF。

三、Weight Normalization 解密

Weight Normalization 是 2016 年由 Tim Salimans 提出的参数化技巧。它把一个权重向量 w 分解为方向 v 和大小 g 两个参数：

w = g · v / ‖v‖2

训练时（需要 Weight Norm）

把方向和大小解耦后，优化地形更平滑：

  • v 控制方向——梯度更新只改变方向，不改变范数
  • g 控制增益——独立学习每个通道的缩放

类比电路：v 是分压电阻比值（决定分压比），g 是总增益（决定放大倍数）。分开调比一起调快得多。

推理时（不需要 Weight Norm）

训练完成后，v 和 g 已经收敛到最优值。推理时可以直接用 w = g · v / ‖v‖ 代替，省去每次前向传播的除法和开方。

PaddleSpeech 的官方代码在 set_state_dict 后会调用 remove_weight_norm()，把权重恢复为普通的 w。

我们的踩坑经历

尝试 1：不做任何处理
→ weight_v/weight_g 是随机值，输出为噪音
尝试 2：set_state_dict → remove_weight_norm → weight_norm
→ remove_weight_norm 用随机 v×g 覆盖了正确的 weight
尝试 3：手动修正 weight_v = weight/‖weight‖
→ weight_g 也未被加载，仍是随机值
尝试 4：同时修正 weight_g 和 weight_v
→ forward 中 weight_norm 的计算方式与手动有微妙差异
✅ 最终方案：先 remove_weight_norm，再 set_state_dict
→ 权重直接写入，不经过 weight_norm，输出完全一致（差异=0.000000）
关键顺序：remove_weight_norm() → set_state_dict()
（先移除再加载，而非先加载再移除）

四、ONNX INT8 量化：精度换速度

神经网络默认使用 FP32（32 位浮点数，IEEE 754 单精度）存储权重。INT8 量化把 32 位浮点映射到 8 位整数：

wint8 = round( wfp32 / scale ) + zero_point

其中 scale 和 zero_point 根据权重分布自动计算：

  • scale = (wmax − wmin) / 255 —— 把 FP32 的范围线性映射到 [0, 255]
  • zero_point = round(−wmin / scale) —— 确保 0 在量化后仍然是 0

为什么不损失精度？

训练好的神经网络权重往往集中在很小的范围（如 [−0.5, 0.5]）。FP32 的 23 位尾数在这种范围下提供了远超需要的精度。INT8 虽然只有 256 个离散值，但在 [−0.5, 0.5] 区间内，量化步长仅为 0.5/128 ≈ 0.004——对于大多数神经网络来说，这个精度足够。

类比：24-bit/192kHz 母带 → 16-bit/44.1kHz CD。绝大多数人听不出区别，但文件小 4 倍。INT8 量化就是神经网络版的「有损压缩」。

动态量化静态量化

# 一行代码完成动态量化
from onnxruntime.quantization import quantize_dynamic, QuantType

quantize_dynamic(
    model_input="g2pW.onnx",       # FP32 原版 (606MB)
    model_output="g2pW_int8.onnx",   # INT8 量化版 (152MB)
    weight_type=QuantType.QInt8,     # 8 位整数权重
)

五、CPU 推理优化策略（无 GPU 的 ARM 设备）

树莓派 5 的 4 核 Cortex-A76 没有 GPU 加速（无 CUDA/OpenCL），也没有 SIMD 矩阵指令（无 AVX-512/SVE）。所有矩阵运算都是标量计算。

优化策略GPU 收益ARM CPU 收益本项目
INT8 量化★★★★★★★★★☆✅ G2P
算子融合★★★★☆★★★☆☆✅ ONNX Runtime
减少数据搬运 (小模型)★★★☆☆★★★★★✅ state_dict
多线程 (2-4 线程)★★☆☆☆★★★★☆✅ 已配置
知识蒸馏 (小模型)★★★★☆★★★★☆待探索

ARM CPU 上最有效的优化不是「算得更快」，而是「算得更少」和「搬得更少」：

  • 算得更少：模型小了 71%，计算量自然减少
  • 搬得更少：模型可以完全放进内存，减少磁盘 I/O
  • 缓存友好：小模型更适配 CPU L2/L3 缓存

source ~/miniconda3/etc/profile.d/conda.sh

# 2. 启动原版 Web 服务（端口 8765）
python ~/tts/tts_web.py

# 3. 或启动量化优化版（端口 8766，推荐）

# http://192.168.3.108:8766  (优化版)
